# ecRad Reference Files

This directory is the expected local home for external ecRad/ecCKD reference
NetCDF files used by the radiative-heating validation gate.

The current manifest is generated by:

```sh
julia --project=test validation/ecrad_reference_manifest.jl
```

Required files:

- `clear_sky_tropical_column.nc`
- `all_sky_tropical_column.nc`
- `rcemip_style_column_subset.nc`

Required variables are listed in
`validation/results/ecrad_reference_manifest.json`. The first acceptance pass
expects layer/interface pressure and temperature, gas profiles, boundary
conditions, reference fluxes, and heating rates. The all-sky case additionally
expects cloud fields, and the RCEMIP-style subset expects a `column` dimension
or variable identifying selected columns.

The hard-threshold comparison gate is generated by:

```sh
julia --project=test validation/ecrad_candidate_schema.jl
julia --project=test validation/ecrad_accuracy_gate.jl
```

The `test` project is used because `NCDatasets` is optional for the runtime
package but required for NetCDF schema and threshold checks.

For each reference flux/heating variable, the same NetCDF file must also
contain a matching RadiativeHeating candidate variable:

- `radiative_heating_lw_up`
- `radiative_heating_lw_down`
- `radiative_heating_sw_up`
- `radiative_heating_sw_down`
- `radiative_heating_heating_rate`

The accuracy gate applies the thresholds from
`validation/ecrad_reference_manifest.jl` to these reference/candidate pairs.
The candidate schema preflight requires every candidate variable to have the
same shape as the corresponding ecRad reference variable before the numerical
accuracy gate is meaningful.
It also computes net downward flux at the first interface as the TOA forcing
diagnostic and at the last interface as the surface forcing diagnostic:

```text
F_net = lw_down - lw_up + sw_down - sw_up
```

Both boundary diagnostics are checked against the hard absolute-error
thresholds in `validation/ecrad_reference_manifest.jl`.

These files are external scientific reference artifacts and should not be
replaced with toy fixtures. If the files are too large for the repository,
store them through the project artifact/data mechanism and materialize them at
these paths before running validation.
